## Tier 1 — 直近fixに残る問題（最優先・自分の修正の不完全/新規バグ）

### T1-1 [HIGH] 検索: サロゲート境界のゼロ幅マッチで以降を取りこぼす（fix4の射程内・新規露出）
- **場所**: `src/core/search/search_engine.cpp` 空マッチ前進（`start = m_end + 1`）と、今回 fix4 で追加した `rc<0` 分岐
- **内容**: ゼロ幅マッチ（`a*`・`\b` 等）の次回開始位置を 1 コードユニット進めるため、直後が BMP外文字（絵文字・CJK拡張）だと上位サロゲートの途中に着地。`PCRE2_UTF` 有効・`NO_UTF_CHECK` 未指定のため `pcre2_match` が `PCRE2_ERROR_BADUTFOFFSET` を返す。今回 fix4 で入れた `rc<0` 分岐は MATCH/DEPTH/HEAPLIMIT 以外を一律 `break`（＝これ以上ヒット無し扱い）するため、**当該位置以降の全マッチ・全置換を静かに欠落**。
- **修正**: 空マッチ前進をサロゲートペア単位に（上位サロゲートなら `+2`）。`rc<0` は `PCRE2_ERROR_NOMATCH` のみ正常終了の `break` とし、それ以外の負値は `truncated=true` で返す。`\b`/`a*` を `"a😀a"`/`"😀😀"` に当てる回帰テスト追加。

### T1-2 [HIGH] confirm_all の「失敗→未読維持」が set_baseline 非原子性で破れる（fix5は不完全）
- **場所**: `src/core/snapshot/snapshot_store.cpp` `set_baseline`（entry変異が先・`objects_.put` が後）／呼び出し `review_flow.cpp:187-197`
- **内容**: fix5 は `review_flow` 層で退避失敗を検査してスキップするが、その下の `set_baseline` は `baseline_hash/mtime/size/unread=false` を**先に**書き換えてから fallible な `objects_.put` を呼ぶ。put が I/O 失敗（ディスク満杯・権限・ネットワークドライブ切断＝CLAUDE.md が名指す破綻系）すると、entry は「baseline_hash=新・baseline_object=旧stale・unread=false」の不整合のまま。confirm_all は err でスキップするが (a) unread が false に落ち未読維持されない、(b) restore_baseline は旧内容を返す一方 baseline_hash は新を主張＝差分/巻き戻し基準が静かに壊れる。`confirm()` 単体も同じ。fix5 の「データを失わない/取消保証」を I/O失敗時に部分的に裏切る。
- **修正**: `set_baseline` を「成功するまで entry を変異させない」順序へ。新値をローカルに組み立て、`put` 成否（or ハッシュのみ記録の分岐）確定後に一括コミット。put 失敗時は entry を一切触らず err。`objects_.put` を失敗させる回帰テスト追加（entry が呼び出し前と完全不変であること）。

### T1-3 [MEDIUM] inline_diff トリムの b側スナップが不正UTF-8で文字を割る（fix2の射程内）
- **場所**: `src/core/diff/inline_diff.cpp` `trim_based_spans` の prefix/suffix スナップ
- **内容**: 今回 fix2 で追加したトリムフォールバックの境界スナップは `a[prefix]` のみで判定し b 側は a 決め打ち。共通prefixの先で a/b は相違するため、不正UTF-8では `b[prefix]` が継続バイトになり得て、`out_b` の区間が new_line のコードポイント途中を指す＝ヘッダが明文で約束する「区間は文字を割らない」不変条件を破る。到達経路: `encoding` の Utf8/Utf8Bom 経路は生バイトを無検査で content にするため、不正バイト列の100KB級行が trim に届く。
- **修正**: prefix/suffix を `pa/pb`・`sa/sb` に分け、各行の自バイトで独立スナップ。`out_a={pa,a.size()-sa}` / `out_b={pb,b.size()-sb}`。

## Tier 2 — 既存コードの高重大度（系統的/原則違反）

### T2-1 [HIGH] UTF-8↔fs::path が Windows ACP 再解釈（非ASCIIファイル名・非ASCIIデータルートで喪失/誤配置）
- **場所**(同根2件):
  - `snapshot/object_store.cpp`・`snapshot_store.cpp`・`index_io.cpp`・`state/state_io.cpp`（パス合成/存在判定/削除を `fs::path(std::string)`/`.string()` で実施）
  - `watcher/resync.cpp:24-29,71,114-115`（`generic_string()` で相対/絶対パス取得）
- **内容**: 契約はパス文字列＝UTF-8 std::string（`object_store.h:37` 明記）。最下層 `atomic_file`/`secure_dir`/`fs_probe` の `to_wide` は `MultiByteToWideChar(CP_UTF8,...)`。ところが上記は `fs::path(std::string)`／`generic_string()` を使い、MSVC STL では **ACP(CP_ACP)** でワイド化/エンコードされる。結果、UTF-8 バイト列が ACP 誤デコードされ、(a) **日本語ファイル名**は resync で baseline キー不一致＝毎回 Created/Removed・Modified 検知崩壊、(b) **ユーザー名に非ASCIIを含む `%LOCALAPPDATA%\pika\`** で index.json/objects/state が別パスへ＝snapshot/state の喪失/誤配置。**日本語専用ツールで実害大。**
- **修正**: UTF-8↔path を util の単一ヘルパに集約（`fs::path(std::u8string(...))` と `p.u8string()`）。`object_store/snapshot_store/index_io/state_io/secure_dir/resync` の全 `fs::path(std::string)`・`.string()`/`.filename().string()` を置換。

### T2-2 [HIGH] revert_all（一括取消）が旧ベースラインを復元せず退避ごと削除（取消が no-op＋復元点喪失）
- **場所**: `snapshot/snapshot_store.cpp:210-233`（`revert_batch`）／`document/review_flow.cpp:202-207`（`revert_all`）
- **内容**: 要件8.3 は「すべて確認済み」の各更新前ベースラインを baseline-replace 退避に保存し**ワンクリックで一括取消**できると保証（`review_flow.h:108-110` も revert_all を「旧ベースライン復元の起点」と明記）。しかし `revert_batch` は同 batch_id の退避を **erase するだけ**で、退避内容を読み出して `set_baseline` で baseline を戻さない。さらに erase 後に `sweep_unreferenced_objects` で旧 object を物理削除。結果、取消後もベースラインは新内容のまま＝**取消が効かず、唯一の復元手段（退避）も破棄**。設計原則1違反。テスト `RevertAllUndoesBatchBaselineReplace` は「退避が消えた件数」しか見ておらず素通り。
- **修正**: `revert_batch` を「退避内容で旧ベースラインを復元してから erase」に。各退避 `s` から `restore_stash(s.hash)` で旧内容を取り出し、`baseline_object=s.hash`・`baseline_hash`・mtime/size・unread を confirm_all 直前へ復元（取り出し失敗時は退避を残す＝安全側）。全復元後に sweep。テストは取消後 `restore_baseline` が旧内容を返すことを assert。

### T2-3 [HIGH] 行差分 dtl(O(N·D)) が編集距離未ガードで実質O(N²)・計算中キャンセル不能
- **場所**: `diff/diff_engine.cpp:119`（`diff_lines`）／`diff/diff_types.h:95`（`max_lines=200000`）
- **内容**: 開始前ガードは bytes/行数/最長行長のみ。本体 `dtl::compose()` は O(NP)＝編集距離 D に対し O(N·D)。両側ほぼ全行相違の20万行で D≈20万→snake 反復が約 8e10 回＝ワーカーでも数分固着。`compose()` は内部キャンセル点を持たず、`compute` は前後でしか cancel を見ないため**走り出すと CancelToken で止まらない**（`cancel_token.h` 自身が「開始前ガードが唯一の対策」と認める）。原則2違反。
- **修正**: 開始前に共通プレフィクス/サフィックス行を剥がし、残差行数（編集距離の上限近似）が閾値超なら `truncated=true` で開始しない分岐を追加。`DiffLimits` に `max_diff_lines`（例 5万）を足す。

## Tier 4 — Low（14件・抜粋）

- **検索**: ReDoS打ち切りが巨大入力/件数上限と同じ `truncated` に集約され区別不能（消費者実装時に `TruncateReason` enum を）／単一 `pcre2_match` は match_limit 到達までキャンセル不能（worker占有・有限時間で返るため low、仕様明文化推奨）。
- **confirm_all**: `freeze_hash` 未設定 target が混じると並行変化スキップ保護をすり抜け未確認内容を baseline 化しうる（confirm_all 経路では空を契約違反としてスキップ推奨）。
- **CLI**: `SetConsoleOutputCP` がパイプ非文字化けを担保するというコメントの因果が不正確（実担保は `/utf-8` リテラル。コメント修正のみ）／未知オプション判定の `a[1]!='\0'`・`a!="-"` がデッド条件（簡約可）。
- **secure_dir**: 既存ディレクトリに owner-only ACL を再適用/検証せず素通り、ACL 失敗時の継承フォールバックが沈黙（要件9.1。ポータブル版で露出余地。WARNログ＋冪等再適用推奨）／`std::string` バッファを `TOKEN_USER*` へ `reinterpret_cast`（x64 実害なしだが規格上UB、`vector<unsigned char>` 推奨）。
- **render**: CSP に `base-uri`/`form-action`/`frame-ancestors` が無く二重壁の片側が非対称（`'none'` 常時付与で防御強化。docs/design.md 6章テンプレ→実装の順）。
- **watcher**: `xxh3_64_lf` の hash 値0をセンチネルに使うため正規化後ハッシュが偶然0だと自己保存抑制が外れ外部変更誤検知（約2^-64・安全側。`optional` で型分離推奨）。
- **TaskRunner**: 停止後 `submit` は「無効なfuture」と謳うが実際は valid+broken_promise（`.get()` が例外）。コメント修正で十分。

## 棄却（9件・主なもの）

敵対的検証で実害が確証できず false_positive 判定:
- decode_auto の lossy で原ファイル静かに破壊 → `mb_to_utf8` は常に `MB_ERR_INVALID_CHARS` 付きで主張の機構が不成立。
- resync 内容ハッシュ無上限で OOM(high申告) → 2倍ピークは事実だが「10MB以上ハッシュのみ」要件の層を取り違え＝high根拠崩れ。
- confirm_all 部分適用で revert_all 可逆性が崩れる(medium) → can_stash 前提で harm chain が成立せず（ただし T1-2/T2-2 が別経路で本質を捉える）。
- inline_diff の budget が歪み形で実セル超過 / tokenize 後破棄の無駄 / CompiledPattern の生ポインタ二重保持 / resync FsEvent.at=0 等 → 事実は成立するが実害なし。

## 修正の実施と再検証（2026-06-14 追補）

確証25件（Tier 1-4）を**全て修正**した。検証＝`ctest --preset x64-core-test` で **gtest 391件全PASS**（従来373＋新規18）・`/W4 /WX` 警告ゼロ。`main_console.cpp`（core-test 非対象）は MSVC `/Zs /W4 /WX` で警告ゼロを個別確認。

### 修正サマリ（Tier 別）
- **Tier 1**: T1-1 検索ゼロ幅マッチのサロゲート前進＋`rc<0` を `NOMATCH` のみ break（他は `truncated`）で取りこぼし解消／T1-2 `set_baseline` を `put` 成功後の一括コミットに原子化／T1-3 `trim_based_spans` の prefix/suffix を a/b 独立スナップ。
- **Tier 2**: T2-1 `util/path_util.h`（utf8_to_path/path_to_utf8）に集約し snapshot/state/secure_dir/resync の CP_ACP 経路を一掃（非ASCIIパス往復テスト追加）／T2-2 `revert_batch` を「退避内容で旧ベースライン復元＋未読へ戻す」に修正（取消が機能）／T2-3 diff に同値行オーバーラップ式の編集距離ガード（`max_diff_lines` 既定5万。全追加/全削除・大ファイル散在編集は通す）。
- **Tier 3**: T3-1 `xxh3_64_lf` の `noexcept` 除去＋`content_hash_lf` で `bad_alloc`→`Result::err(Io)`（terminate 回避）／T3-2 save 順序不変条件をヘッダ明文化／T3-3 共有 object サイドカーを退避単位に分離（復元メタの取りこぼし防止）／T3-4 終了コード/メッセージ決定を `decide_console_outcome` に切り出し cli_parser_test で検証化。
- **Tier 4（14件）**: CSP に `base-uri`/`form-action`/`frame-ancestors` を常時 `'none'`（design.md 6章先行）／hash 値0センチネルを `HashProbe = optional<uint64_t>` で型分離／`secure_dir` 既存dir・継承フォールバックへ owner-only DACL 冪等強制適用＋`reinterpret_cast` を `vector<unsigned char>` に／`TaskRunner` future 契約コメント是正／cli デッド条件簡約・SetConsoleOutputCP コメント因果是正 ほか。

### 方針上の判断（指摘の文言どおりにしなかった2点）
- **T2-3 の編集距離近似**: レビュー推奨の「共通先頭/末尾行の剥がし」は大ファイルの散在編集を過剰に打ち切るため、**同値行オーバーラップ方式**（O(N) ハッシュ・両側非共通行が閾値超のときのみ truncate）を採用。pika の本質＝編集レビューを過剰に弾かない。
- **D4 freeze_hash**: 推奨の「空 freeze_hash を confirm_all で自動スキップ」は `confirm()` 互換の「freeze 無し確認」用途と既存テストを壊すため、**契約をヘッダに明文化**（呼び出し側が freeze_hash を設定する責務）する方針にした。

### 修正の敵対的再検証（implement→review）
9バッチを独立エージェントで再レビュー（修正が「正しく・完全に・退行なく」効いているか、修正コード自体に新バグがないか）。結果＝**全9バッチ correctly_fixed=true / complete=true / regression=false**。新規問題は1件のみ（LOW）＝secure_dir の戻り値の意味が変わったのにヘッダ doc が旧契約のままという doc/code 不整合 → **同日修正済み**（戻り値＝「DACL強制適用の成否」で保全継続の中断条件に使わない旨を明記）。

### 残課題（本修正の射程外・既知）
- T3-2 の object削除↔index.json保存の非原子性窓は `SnapshotStore` ジャーナリング相当が必要で別タスク（不変条件は明文化済み）。
- 秘匿性失敗の上位通知（`secure_dir` の戻り値2種の分離・通知バー連携）は DocumentController/UI 実装段で配線する。
