## 1. 技術スタック（決定）

> **【Tauri フル刷新フェーズで改訂・2026-06-21】** 本章の技術スタックは
> `docs/specs/2026-06-20-tauri-rewrite-architecture-design.md`（design doc）2章へ全面差し替えた。
> 機能要件（2章以降）はそのまま正典として引き継ぐ。本章の旧 wx/C++ スタックは
> 「載せ替え前の設計」として末尾に保全する（履歴を失わない）。

軽量さを最優先基準とした再評価により、UI フレームワークを Web 全描画＋CSS 変数トークンへ刷新する
（VSCode/Obsidian 同等の質感を wx 版 9.6MB と同等以下の軽さで両立。Tauri スパイク実測 8.6MB）。

| 項目 | 決定 | 理由（要点） |
|---|---|---|
| アプリフレームワーク | Tauri 2 | OS の WebView2 を使い Chromium を同梱しない＝極小バイナリ。Rust backend |
| backend 言語 | Rust（stable 1.96+） | ネイティブ・単一ランタイム・FFIグルー不要 |
| UI 描画 | HTML/CSS/TypeScript（全Web） | OSコントロール非依存＋CSS変数トークンで質感 |
| エディタ部品 | CodeMirror 6 | 軽量・モジュラー・Markdown 得意・MIT |
| プレビュー/差分描画 | 権限ゼロの別 WebView（custom protocol 直配信＋CSP） | 未信頼文書をアプリシェル（Tauri API）から物理隔離。CVE-2024-35222（同一オリジン iframe invoke 例外）を回避 |
| Markdown 変換 | comrak（GFM・`unsafe_=true` で raw HTML 透過） | Rust・高速・GFM。raw HTML は最終段で ammonia サニタイズ |
| HTML サニタイズ | ammonia（ホワイトリスト・最終段固定） | `<script>`・on*・`javascript:`・`<iframe>/<object>/<embed>/<base>/<meta>` 除去 |
| 差分エンジン | similar（Myers）＋文字単位フォールバック（unicode-segmentation） | 日本語等の語境界不成立行を grapheme 単位へ |
| 検索/置換の正規表現 | fancy-regex（第一候補・純Rust）／pcre2（機能不足時のみ） | 後方参照・キャプチャ参照・Unicode文字クラス。ReDoS タイムアウト必須 |
| ファイル監視 | notify crate＋自前合成層／オーバーフロー検知不可なら windows crate `ReadDirectoryChangesW` 直叩き | デバウンス/合体・rename正規化・自己保存抑制・オーバーフロー再同期・ポーリングは自前 |
| 圧縮 / ハッシュ | zstd crate / twox-hash（XXH3） | スナップショット圧縮・重複排除 |
| 設定パース / エンコーディング | toml crate / encoding_rs | settings.toml 読取専用 / BOM・UTF-8・UTF-16・Shift_JIS 往復 |
| 単一インスタンス | 自前 named pipe を既定実装（`\\.\pipe\pika-<SID>`・ユーザー限定DACL・`PIPE_REJECT_REMOTE_CLIENTS`・受信≤数KB・JSONスキーマ検証） | 信頼境界 |
| CLI | 二段構成: `pika-cli.exe`（console subsystem・--help/--version/引数検証/終了コード）＋`pika.exe`（windows subsystem・GUI） | GUI subsystem 単体ではシェル制御戻りを保証できない |
| エラー型 / ビルド | thiserror の `Result<T, PikaError>` / cargo workspace＋Vite＋Tauri bundler | コア公開API は Result 方式 |
| 依存監査 | `cargo audit` を CI ゲート | comrak/ammonia/Mermaid/KaTeX 等の CVE 追跡 |

WebView2 Runtime の前提：全Web化により「不在時もエディタ/ツリーは動く」は構造的に不可能。
**WebView2 を必須前提とし、不在/破損時は Tauri 起動前に最小ネイティブダイアログで導入案内して終了する**
（design doc 18章。詳細は 2.3 改訂）。

アーキテクチャ原則（design doc 1章・3章。旧「ネイティブ優先」を置換）：

* **全Web描画**。UI（ツリー/タブ/エディタ/通知/検索）は HTML/CSS/TS で実装し、CSS 変数トークンで質感を作る
* **未信頼文書は権限ゼロの別 WebView で隔離する**（文書由来 HTML/Markdown を Tauri API から物理分離）
* **コアは UI を知らない**：`pika-core`（Rust）は Tauri/wry/WebView を知らない。橋渡しは Tauri command/event のみ
* レンダラー（comrak→ammonia）、差分（similar）、監視、CLI、スナップショットは疎結合な Rust モジュール

## 2. 非機能要件

### 2.1 性能目標

計測条件：Windows 11、基準機（4物理コア・ベース2.5GHz以上のx64／8GB RAM／NVMe SSD、電源プランは「高パフォーマンス」固定）で計測する。基準ファイルは特記なき限り10KBのMarkdownファイル1つだが、メモリは下表の各行に併記した代表ワークロード別の条件で計測する。「コールド起動」はアプリのプロセスが存在しない状態からの起動を指す。メモリはWebView2プロセス群を含むアプリ全体のワーキングセット合計とし、瞬間値（コールド起動直後）だけでなく常駐型ツールとしての定常状態（長時間稼働後）も対象とする。計測手順：各項目を10回計測し、目標は中央値・上限はp95で判定する（メモリはピークワーキングセットの中央値）。計測はリリースビルドで行い、手順と結果を `docs/perf.md` に記録する。

| 項目 | 上限（受け入れ基準） | 目標 |
|---|---|---|
| コールド起動 → ウィンドウ表示 | 0.5秒 | 0.3秒 |
| 起動 → 指定ファイルが表示・編集可能 | 1.0秒 | 0.5秒 |
| プレビュー初回表示（WebView2初期化込み） | 2.0秒 | 1.0秒 |
| 編集中のプレビュー更新（デバウンス込み） | 300ms | — |
| タブ/モード切替後のプレビュー・差分の再表示 | 300ms | 150ms |
| 外部変更 → エディタ/ツリーへの反映 | 500ms | — |
| メモリ：最小（WebView2未起動。`.log` 等プレビュー非対象ファイル、または `.md` をソースモードに明示切替して開いた状態。6.1の既定〔プレビューのみ〕ではない点に注意） | 60MB | 40MB |
| メモリ：`.md`を既定（プレビューのみ）で開いた直後の常用状態（WebView2起動済み） | 350MB | 250MB |
| メモリ：プレビューを一度表示後ソースのみに戻したアイドル状態（WebView2サスペンド/解放後） | 250MB | 180MB |
| メモリ：代表ワークロード（1000ファイルのフォルダを開き、10MB級ファイルを含む5タブを開いた状態） | 450MB | 350MB |
| メモリ：定常状態（8時間連続稼働＋外部変更1000回処理＋プレビュー50回更新後） | コールド時実測比 +20% 以内（リーク非発生） | — |
| 配布サイズ（単一exe＋同梱アセット） | 30MB | — |

* 200msを超えるUIブロックを発生させない。ファイル読み込み・差分計算・スナップショット処理・Markdown変換は非同期またはワーカースレッドで行う
* 読み込みに時間がかかるファイル（2.2の段階制対象等）は進捗表示を行い、その間もUIは操作可能とする
* 常駐運用での長時間安定性：数時間〜数日の連続稼働、外部変更の連続処理、プレビューの反復更新を経ても、ワーキングセットが上表の定常状態上限を超えて単調増加しない（メモリリーク・断片化を発生させない）。WebView2の再ナビゲート、md4c変換結果の差し替え、watcherイベントの連続処理を主要なリーク監視対象とする（連続稼働ソークテストは設計書13章）

### 2.2 巨大ファイル・長行の段階制

メジャーエディタの標準パターン（段階制＋自動オフ＋閾値設定可。VS Code 20MB / Notepad++ 200MB 等）に準拠する。

> **【Tauri 刷新 TBD・2026-06-21】** CodeMirror 6 はインメモリ文書モデルで Scintilla ほど巨大ファイルに
> 堅くない（design doc 8章）。**第2段階・上限の数値は sprint 6 の CM6 実測で確定**し本表を改訂する。
> 第1段階（10MB）は「AI出力の単一行巨大 JSON/JSONL の編集が後退しない」中心シナリオ死守の観点で
> 暫定維持。実測で 10MB を下回って劣化するなら中心体験後退として別途要件改訂を提案する。
> 推測値を canon に書き込まないため、確定前の第2段階/上限は **TBD（実測 sprint で確定）** とする。

| 段階 | 既定閾値 | 挙動 |
|---|---|---|
| 第1段階 | 10MB以上（暫定維持・実測ゲートで確認） | プレビュー・差分・ベースライン内容の保存・シンタックスハイライトを自動オフ。通知バーで明示し、各機能を個別に手動再有効化できる。閲覧・編集・検索・保存は通常通り。未読判定はハッシュのみ記録（9.2）で維持する（差分は見られない）。内容保存の境界は9.2を正とし「10MB未満のみ内容保存／10MB以上はハッシュのみ」に統一する（ちょうど10MBはハッシュのみ）。閾値を跨ぐ遷移時の挙動は9.2、退避が取れない場合の破壊的操作は7.3・8.3、確認済みによるハッシュ更新は8.1に従う |
| 第2段階 | **TBD（sprint 6 の CM6 実測で確定。旧値 200MB超）** | 読み取り専用ビューアモード（閲覧・検索のみ。編集・保存・置換は不可）。Rust ストリーミング range 読取＋仮想化ビューア（design doc 8章） |
| 上限 | **TBD（sprint 6 の CM6 実測で確定。旧値 2GB超）** | 開かず、明確なエラーメッセージを表示する |

* 行長ガード：1行10万文字を超える行を含むファイルは、サイズ閾値と独立にシンタックスハイライト・折返しを自動オフする（AI出力の単一行巨大JSON/JSONL対策）
* レンダリング暴走ガード：サイズ閾値以下でも、プレビュー（Mermaid/SVG/HTML）と画像表示にはリソース消費の上限を設ける。既定で、画像の総ピクセル数6000万px、SVGの展開ピクセル数8000万px相当・要素数5万、HTMLのレンダリングタイムアウト10秒を超える入力はレンダリング/デコードを行わず、通知バーで「既定のブラウザ/アプリで開く」へ誘導する（設計原則「固まらない」。AI出力＝非定型を主対象とするため既定オン）。上限はすべて設定で緩和できる
* すべての閾値は設定で変更可能とする

### 2.3 動作環境

* 対応OS：Windows 11（x64ビルドのみ）。ARM64機ではx64エミュレーションでの動作を妨げない（検証対象外）
* WebView2 Runtime（**Tauri 刷新で改訂・2026-06-21・design doc 18章**）：全Web化によりアプリシェル全体が
  WebView2 上にあるため、不在/破損/最低版未満ではウィンドウすら描けず、旧「エディタ/ツリー/監視は動く」は
  構造的に満たせない。**WebView2 を必須前提とし**、不在/破損時は **Tauri 起動前に最小ネイティブダイアログ
  （Win32 MessageBox・WebView 非依存）**で Evergreen Runtime の導入/更新を案内して終了する。
  実装は `src-tauri/src/webview2.rs`（EdgeUpdate Evergreen GUID の `pv` キー検出）。
  * （保全）旧要件: 不在・破損・最低版未満でもアプリは起動し、プレビュー・差分表示のみ無効化して Runtime の
    導入・更新案内を表示する（エディタ・ツリー・監視は動作する）。← 全Web化で達成不能のため上記へ改訂。
* UI言語：日本語のみ。文言はリソースとして一元管理し、将来の多言語化の余地を残す

### 2.4 ネットワーク・プライバシー

* 外部通信は、プレビューのリモートリソース取得（6.2）のみとする。これは**既定でオフ（オプトイン）**であり、ユーザーが許可した場合に限り発生する。文書を開いただけでは外部通信は一切起きない
* テレメトリ・使用状況の送信・更新確認を含め、上記以外のネットワーク通信を一切行わない
* 開いたファイルの内容・パスを外部に送信しない

## 3. CLI仕様（エージェント向けAPI）

コマンド名・実行ファイル名は `pika` とする。

### 3.1 コマンド

| 呼び出し | 挙動 |
|---|---|
| `pika <パス>...` | ファイル/フォルダを開く。ファイル複数指定はタブで開く。フォルダ指定は1つまで（単一フォルダ方針。複数指定時はエラー）。フォルダ＋ファイル同時指定はフォルダを開いてそのファイルをアクティブにする |
| `pika`（引数なし） | 前回状態を復元して起動。起動済みならウィンドウを前面化する |
| `pika -g <file>:<行>[:<桁>]` | 指定位置にカーソルを置いて開く（VS Code互換記法）。この場合はソースモードで開く（6.1のモード記憶は更新しない） |
| `pika --help` / `pika --version` | コンソールに出力して終了する |

* `-g` 行指定構文のパース規則：パス末尾から `:<整数>`（さらに任意で `:<整数>`）を桁・行として剥がし、残りをパスとして解釈する。先頭のドライブレター直後のコロン（`C:\...`）は分割対象にしない。桁を省略した場合は行頭にカーソルを置く。行・桁が整数でない場合は位置指定を無視してファイルを開く。行番号がファイル行数を超える場合は最終行に置く

### 3.2 動作規則

* シングルインスタンス：起動済みなら既存ウィンドウで開いて前面化する。ウィンドウは常に1つ（複数ウィンドウは初期版でやらない）。起動レース対策として、名前付きパイプの作成（`CreateNamedPipe`）の成否を原子的なロック獲得とし、獲得した1プロセスのみがサーバー兼ウィンドウになる。獲得に失敗したプロセスはクライアントとして引数を転送して終了する（サーバー公開はウィンドウ表示前に完了させる。詳細は設計5.1）
* 起動中に別フォルダが指定された場合はフォルダを切り替える。未保存変更があるタブが存在する場合は確認を挟む
* フォルダを開いている状態でフォルダ外のファイルが指定された場合、フォルダは維持したままタブで開く。ツリーには表示しないが、監視・未読・差分・退避は単体ファイルの規則（9.1）で同等に機能する
* 相対パスは呼び出し元プロセスのカレントディレクトリ基準で解決する。起動中インスタンスへ転送する場合は、クライアント側プロセスで絶対パスに正規化してから転送する（サーバー側のカレントディレクトリに依存させない）
* 存在しないファイルパスは「保存時に作成される新規タブ」として開く。存在しないフォルダはエラー
* 終了コード：0＝受理、非0＝エラー（不正な引数、存在しないフォルダ等）。GUI起動後は即座に制御を返す（`--wait` は初期版では持たない）。引数の検証はウィンドウ生成・転送の前に同期実行し、検証失敗時はGUIを起動せず非0で即終了する。**転送時の終了コード0は「リクエストを受理した」ことを表し、ファイルが実際に開かれた保証ではない**（ユーザーがフォルダ切替の確認をキャンセルしても受理＝0。エージェントはこの意味で扱う）
* IPC（引数転送）の信頼境界：名前付きパイプ `\\.\pipe\pika-<ユーザーSID>` は作成者ユーザーのみ許可するDACLと `PIPE_REJECT_REMOTE_CLIENTS`（遠隔接続拒否）で保護する。受信は最大数KBで打ち切り、JSONスキーマ検証に失敗したデータは破棄してログに残す。受理する操作はパス引数のオープンに限定する（設計3章 core/ipc）

### 3.3 エクスプローラー統合

* `.md` / `.markdown` / `.html` / `.htm` の「プログラムから開く」候補に登録する
* ファイル・フォルダの右クリックメニューに「pikaで開く」を追加する
* いずれもインストール時に登録し、オプトアウトできる。登録先は `HKCU\Software\Classes` 配下（拡張子の `OpenWithProgids` と `shell\pika\command`）とし、管理者権限を要しない
* ポータブル版はエクスプローラー統合を登録しない（インストール版限定）。これにより、ポータブル版を実行フォルダごと削除しても壊れたパスを指す関連付けの残骸が生じない
* アンインストール時は登録した `HKCU\Software\Classes` 配下のキー（`OpenWithProgids`・`shell\pika`）を完全に削除し、「プログラムから開く」候補や右クリックの残骸を残さない

### 3.4 受け入れ基準（抜粋）

* 起動済みの状態で `pika foo.md` を実行すると、呼び出しプロセスは即座に終了し、既存ウィンドウに foo.md が開かれて前面化する
* `pika -g doc.md:120` で120行目が表示範囲に入り、カーソルが該当行に置かれる
* 未起動の状態で `pika <フォルダ>` を実行してから、2.1の性能目標時間内にツリーとエディタが操作可能になる
* 起動中インスタンスへ `pika <別フォルダ>` を転送し、ユーザーが未保存タブの切替確認をキャンセルしても、呼び出しプロセスは終了コード0（受理）を返す
* `pika --version` / `pika --help` をパイプ・リダイレクト（例：`pika --version > v.txt`）で実行しても出力が欠落・文字化けせず取得でき、出力後にシェルへ制御が戻る

## 4. ファイルツリーと未読表示

### 4.1 ツリー表示

* フォルダ先行・自然順ソート（`file2` が `file10` より前）。ソート順の設定は持たない
* 隠しファイル・ドットファイルも表示する（特別扱いなし）
* 既定の除外リスト：`.git`・`node_modules` はツリー非表示かつ監視対象外とする。除外リストは設定で編集できる（2.1の性能目標を守るための要件でもある）
* ファイル種別をモノトーンの線アイコンで示す（フォルダ／コード／データ／設定／スクリプト／画像／テキスト。詳細 `docs/ui-design.md` 6章）。ツリーペインは収納/引き出しトグルで折りたためる
* シングルクリックでファイルを開く
* 操作：F2＝リネーム、Delete＝削除（ごみ箱へ移動）、右クリックメニュー＝開く／既定のブラウザで開く／差分を見る／確認済みにする／確認済み時点に戻す／退避した内容を復元／（フォルダ）このフォルダ配下を一括確認済みにする／新規ファイル／新規フォルダ／リネーム／削除／エクスプローラーで表示／パスをコピー
* ファイル単体で起動した場合（フォルダ未指定）はツリーペインを表示しない。ツリー非表示時（単体ファイル・フォルダ外ファイル）でも、差分・確認済みにする・確認済み時点に戻す・退避した内容を復元の各操作には、タブバー右端のボタン群またはメニューから等価に到達できる（11.1）

### 4.2 未読の規則

* 外部プロセスによる内容変更・新規作成があったファイルに未読バッジを付ける（UIの視覚表現は「差分ありマーク」＝変更 ±／新規 ◆／削除済み 取り消し線。ツリー・タブ共通。`docs/ui-design.md` 5章）
* pika自身の保存では未読を付けない。自己保存か外部変更かの判定は保存後の内容ハッシュ一致を必須条件とする（時刻窓は補助的な安全弁に過ぎない。詳細は7.1・設計5.2）
* 子孫に未読があるフォルダにもバッジを付ける（折りたたみ中でも気づける）。フォルダ自体の未読（子孫からの伝播）と、ファイル自身の未読は区別できる表示にする（詳細は5.3・設計10章）
* リネーム・移動されたファイルは未読・ベースライン・退避を引き継ぐ。引き継ぎ規則：リネーム先に既存エントリがある場合は移動元で上書きする。相互スワップ（A↔B）は一時退避を経てアトミックに付け替える。短時間の往復（A→B→A）など対応付けが確定できない場合は最終的なディスク内容で未読を再判定する。watcherのrename旧名/新名ペアが成立しない場合（クロスディレクトリ移動・片側欠落・上書きrename）は安全側に倒し（旧名単独＝削除、新名単独＝新規、上書きrename＝内容変更）、引き継ぎに失敗したエントリはベースライン・退避を旧キーで孤立保全して失わない（90日GCに委ねる。詳細は7.2・設計5.2）
* 未読の解除は「確認済みにする」操作で行う（8章で定義）
* 未読はセッションをまたいで保持する。アプリ終了中に変更されたファイルも、次回起動時に既読ベースラインとの比較で未読として表示する

### 4.3 受け入れ基準（抜粋）

* 外部プロセスがファイルを変更すると、500ms以内にツリーへバッジが表示される
* 折りたたまれたフォルダ内のファイルが変更された場合、親フォルダにバッジが表示される
* pikaで編集して保存した場合、バッジは付かない
* アプリを終了し、その間に外部変更されたファイルが、次回起動時に未読として表示される

## 5. エディタ・タブ・検索・保存

### 5.1 編集

* 標準ショートカット（Ctrl+Z/Y/X/C/V/A 等）。Undoは無制限（セッション内のみ。永続Undoは持たない）。外部変更の自動リロード（7.2）もUndo対象とし、1回のリロードを1つの取り消し単位として直前の自分の状態へCtrl+Zで戻せる（リロード自体はdirtyにしない）。これは「確認済み時点に戻す」（8.3。ディスク上のベースライン基準）とは別物
* タブ・インデント：原文のタブ/スペースをそのまま保持し、保存時に正規化しない。Tabキーはタブ文字を挿入する（スペースへ展開しない）。タブ幅の既定は4桁。これらはYAML/JSON/TOML/Markdown等インデントが構文的に意味を持つファイルを壊さないための保守的既定であり、いずれも設定（10.3）で変更できる
* 行番号を表示する。折返しはトグル可能で、既定は `.md`/`.txt` がオン、その他はオフ
* シンタックスハイライトは対応ファイル種別分のScintillaレクサーを使用する（md / html / json / yaml / toml / xml / ini / sh / ps1 / bat 等。csv はScintillaに標準レクサーがないためプレーン表示で可）。未知の種別はプレーンテキスト表示
* `.markdown` は `.md` と、`.htm` は `.html` と同等に扱う（プレビュー・関連付け・モード記憶を含む）
* 2.2の巨大ファイル段階制・行長ガードに従う

### 5.2 エンコーディングと改行コード

* 読み込み時に自動判定する：(1)BOMを最優先（UTF-8 BOM / UTF-16 LE/BE）、(2)BOMなしは候補（UTF-8 / Shift_JIS）を順にデコードして妥当性を検査、(3)いずれも妥当でなければUTF-8として開き警告を表示する。誤判定に備え、「表示」メニューのエンコーディング項目から「エンコーディングを指定して開き直す（Reopen with Encoding）」「現在のエンコーディングを変更（次回保存時に適用、Change Encoding）」を提供する（設計5.7・`docs/ui-design.md` 7章）
* 保存時は元のエンコーディング・BOM・改行コードを維持する（勝手に変換しない）。現在のエンコーディングで表現できない文字（例：Shift_JISのファイルに貼り付けたUTF-8専用文字・絵文字）を検出した場合は保存を中断し、通知バーで［UTF-8で保存／該当文字を確認／キャンセル］を提示する（無確認の不可逆変換・文字欠落は行わない）
* 新規ファイルの既定はUTF-8（BOMなし）・LF。設定で変更可能
* 現在のエンコーディングと改行コードは「表示」メニューに表示する（ステータス右下には置かない。`docs/ui-design.md` 7/9章）
* 既存の混在改行（CRLFとLFが混在するAI出力）は保存時に原文の改行をそのまま維持する（統一しない）。改行コードの表示（「表示」メニュー）は混在時に「混在（CRLF/LF）」と示す

### 5.3 タブ

* 複数タブ。未保存変更のあるタブにはマークを表示する
* タブにも未読インジケーターを表示する（ツリーと同じ規則。単体ファイル・フォルダ外ファイルでも未読が分かる）
* タブのバッジは未読・未保存・削除済みが重畳しうる。表示優先順位は 削除済み ＞ 未保存 ＞ 未読 とし、色だけに依存せず記号・位置でも区別できるようにする（詳細は設計10章）
* タブが画面幅を超える場合はスクロール＋全タブ一覧のドロップダウンを提供する。一覧の各項目にも未読・未保存・削除済みの状態マークを表示し、右端に隠れた未読タブがあることをタブバー端のバッジで示す
* Ctrl+Tabで切替、Ctrl+W・中クリックで閉じる
* 未保存のタブを閉じるときは確認する（保存／破棄／キャンセル）

### 5.4 検索・置換

* ファイル内検索（Ctrl+F）：インクリメンタル検索、大文字小文字区別、単語単位、正規表現、ヒット件数表示、全ヒットのハイライト
* ファイル内置換（Ctrl+H）：1件ずつ／全置換。正規表現置換（キャプチャ参照）に対応する。正規表現は後方参照・キャプチャ参照・Unicode文字クラスをサポートするエンジンを用いる（Scintilla内蔵の正規表現は機能が貧弱なため不採用。具体エンジンは設計で確定）
* 検索はモードに応じて切り替わる：**差分ON、またはソース／分割ではエディタ（テキスト）検索**、**プレビューのみ（差分OFF）ではプレビュー内テキスト検索**（WebView2の検索機能）をCtrl+Fで提供する。置換はソース・分割でのみ使用できる（差分は読み取り専用）
* 巨大ファイル・長行（2.2の対象）での検索・全置換はUIをブロックせず、キャンセル可能な処理として進捗を表示する。第2段階（**TBD・sprint 6 の CM6 実測で確定。旧値 200MB超**＝2.2の連動数値）の読み取り専用モードでは置換を無効化する
* フォルダ横断検索（grep）は初期版では持たない

### 5.5 保存

* 通常保存（Ctrl+S）、名前を付けて保存。自動保存は持たない
* 保存時に外部変更との衝突がある場合の挙動は7章に従う

### 5.6 受け入れ基準（抜粋）

* Shift_JIS・CRLFのファイルを開いて編集・保存しても、エンコーディングと改行コードが変わらない
* Shift_JISと誤判定されたUTF-8ファイルを「エンコーディングを指定して開き直す」で正しく再表示できる
* Shift_JISのファイルに同コードで表現できない文字を入れて保存しようとすると、保存が中断し選択肢（UTF-8で保存／該当文字を確認／キャンセル）が提示される（無確認の文字欠落が起きない）
* 正規表現＋キャプチャ参照で全置換ができる
* 新規タブを保存するとUTF-8（BOMなし）・LFで書き込まれる
* タブが画面幅を超えても、全タブ一覧から目的のタブを選べ、隠れた未読タブの存在がバッジで分かる
* マウスを使わずキーボードのみで「ファイルを開く→プレビュー→差分→確認済み」が完了でき、差分の追加/削除がモノクロ（記号）でも判別でき、ハイコントラストモードでも判読できる

## 6. プレビュー

### 6.1 表示モード

* `.md` / `.html` のタブは「ソースのみ／分割（エディタ＋プレビュー）／プレビューのみ」の3モードをワンクリックおよびショートカットで切替する。**差分は独立した ON/OFF トグル**とし、ソース/分割/プレビューのどのモードにも重ねられる（差分の中身は常にテキスト＝ソースベース）。プレビュー＋差分ONは自動で分割になり「左＝レンダリング、右＝テキスト差分」を表示する。レンダリング済みプレビュー上の差分は持たない（14章）。視覚詳細は `docs/ui-design.md` 8章
* 初回の既定はプレビューのみ。以後は最後に使ったモードをファイル種別ごとに記憶し、次回開くときの既定にする。初回既定モードは設定（10.3「初回既定モード」）で変更でき、軽量起動を重視するユーザーはソースのみを既定にできる（看板60MB〔2.1〕はWebView2未起動のソースモード時の参考値）
* 分割時はエディタ⇔プレビューのスクロールを双方向に同期する（行ベースの近似で可）。ただしHTMLプレビューとSVG表示は同期対象外（独立スクロール）とする（pikaが生成しないHTML／画像レンダリングには行マッピングを注入できないため。詳細は設計6章）
* `.svg` もプレビュー対象とする（WebView2で画像としてレンダリング。モード切替・初回既定は `.md`/`.html` と同じ規則。ソースはXMLテキストとして編集できる）

### 6.2 Markdownプレビュー

* md4cによるGFM準拠（テーブル・タスクリスト・打消し線・自動リンク）。Markdown内HTMLに対応する
* コードブロックのシンタックスハイライトを行う（ハイライトライブラリを同梱し遅延読み込み）
* Mermaid・数式（KaTeX）は同梱・遅延読み込み・オフライン動作。各ブロックを個別にレンダリングし、構文エラーまたはタイムアウト（1ブロックあたり約1秒）時は当該ブロックを元のコードブロック表示へ戻してエラーマークを付ける。失敗が複数あれば件数を通知バーで知らせる
* ローカル画像（相対パス）を表示する。参照先が存在しない場合は壊れた画像のプレースホルダを表示し（黙って空白にしない）、AI出力の壊れた参照だと分かるようにする
* リモートリソース（http画像等）は**既定で読み込まない（オプトイン）**。外部参照を検知したら通知バーで「この文書は外部リソースを参照しています。許可しますか」を出し、許可した場合のみ取得する。許可状態は設定（10.3）で既定値を変更できる（2.4のプライバシー方針と整合）
* 相対パスの `.md` へのリンクはpika内で開く。それ以外のリンクは既定のブラウザで開く。リンク先の `.md` が存在しない場合は新規空タブを作らず、エラーマーク/通知で知らせる
* セキュリティ境界：プレビュー描画では、pikaに同梱した信頼済みJavaScript（Mermaid・KaTeX・コードハイライト）のみを実行し、ユーザー文書（Markdown／HTML）に由来するJavaScriptは一切実行しない。文書由来のHTMLはサニタイズ（`<script>`・イベント属性・`javascript:` URL・`<iframe>`/`<object>`/`<embed>`/`<base>` 等の除去）したうえで、CSPにより `script-src` をpika同梱アセットのみに限定して表示する（実装は設計6章）。計画案7章・14章の「任意JavaScript実行なし」はこの方針（同梱の信頼済みJSは別扱い）を指す

### 6.3 HTMLプレビュー

* WebView2で表示し、JavaScript実行は無効化する（`IsScriptEnabled=false`、ナビゲーション前に設定）。pikaのMarkdownプレビュー・差分・SVG表示では同梱の信頼済みJSのみが動く（6.2のポリシー）ため、ユーザー文書のHTMLプレビュー（JS無効）とは表示内容ごとにWebView2のJS設定を切り替える（二系統。切替の直列化は設計6章）
* インラインCSS（`<style>`・style属性）を完全にレンダリングする
* `<script>` タグ・Tailwind CDN参照を検知した場合、通知バーで「JSを使用するHTMLのため表示が崩れる可能性がある」ことを伝え、「既定のブラウザで開く」ボタンを表示する
* HTMLプレビューも2.4のネットワーク方針の対象とする。外部 http(s) リソース（画像・CSS・フォント・`<meta refresh>`/preload 等）は**既定で遮断**し、ローカル相対参照（`.css`・画像）は仮想ホスト経由でWebView2に解決させる（`file:///` 直接ナビゲートはしない。実装は設計6章）
* リモートリソースの取得はMarkdownと同じく既定オフ（オプトイン）で、外部参照検知時に同じ通知バー導線で許可できる
* プレビュー内のリンクはMarkdownと同じ規則とする：相対パスの `.md`/`.html` はpika内で開き、それ以外は既定のブラウザで開く（プレビュー内でのページ遷移はしない）
* 壊れた相対参照（存在しない画像・リンク切れ・存在しない `.md`/`.html`）の挙動は6.2と同一（プレースホルダ表示／新規空タブを作らずエラーマーク）
* 「既定のブラウザで開く」はJS検知時の通知バーに限らず常設とする：メニュー、ツリー右クリック、タブバーのボタン群から常に実行できる（計画案6章のMVP機能）

### 6.4 受け入れ基準（抜粋）

* GFMのテーブル・タスクリスト・Mermaid・KaTeX数式を含むMarkdownが崩れずに表示される
* インライン`<style>`を持つ自己完結HTMLが、ブラウザ表示と同等の見た目でレンダリングされる
* `<script>`を含むHTMLを開くと通知バーが表示され、ボタンから既定のブラウザで開ける
* プレビュー未使用のセッションではWebView2プロセスが起動していない（2.1のメモリ要件と対応）

## 7. 外部変更の反映と衝突処理

### 7.1 検知

* `ReadDirectoryChangesW` によるフォルダ再帰監視（4.1の除外リストを適用）。ファイル単体モードでは親フォルダを監視して対象ファイルのみフィルタする
* 連続書き込み対策としてデバウンスする（目安100ms）。エージェントのストリーム書き込み中の中途半端な内容を読まないため、直近変更からデバウンス分の静穏期間を待ち、かつmtime/サイズが連続イベント間で安定したことを確認してから内容を確定読みする。書き込みロック中（共有違反）は短間隔でリトライし、上限を超えた場合はベースライン確定を保留する（中途内容で確定しない）
* アトミック置換（一時ファイル書き込み→rename方式の保存）も変更として正しく検知する
* 多数ファイルの同時変更（AIエージェントが配下の数十〜数百ファイルを一斉に生成・書換）が主シナリオであることを前提とする。`ReadDirectoryChangesW` のカーネルバッファが溢れた場合（`ERROR_NOTIFY_ENUM_DIR`）は、該当監視ルートを全再列挙し、mtime/サイズ→ハッシュ比較で未読・ベースラインを再同期して取りこぼしを防ぐ（監視バッファは既定64KB・設定可）
* `ReadDirectoryChangesW` が機能しない環境（ネットワークドライブ/UNC/一部クラウド）では、定期ポーリング（既定5秒・設定可。9.2のmtime+サイズプレスクリーンを再利用）へフォールバックし、監視機能を縮退させず維持する（12.1。F5で即時の全体再スキャンも提供）
* pika自身の保存による変更イベントは無視する。判定は保存後の内容ハッシュ一致を必須条件とし（現ディスク内容のハッシュが保存後ハッシュと一致する場合のみ自己イベントとして1回だけ消費）、時刻窓は誤抑制を防ぐ補助的な安全弁としてのみ用いる。窓を超過してもハッシュが一致すれば自己保存として抑制し、内容が異なれば外部変更として扱う（詳細は設計5.2）

### 7.2 反映（未保存変更がない場合）

* 自動リロードし、スクロール位置・カーソル位置・プレビュー表示位置を維持する（行がずれた場合は近似位置）。リロードはUndo境界を1つ作って実行し、内容差し替え後もdirtyにしない（5.1）
* リネームはタブを新パスに追従させる（未保存変更も維持）。watcherのrename旧名/新名ペアが成立しない場合は、新名を特定できれば追従し、旧名単独（移動先が監視外等）なら「削除済み」表示としてタブは閉じない（4.2の正規化規則に従う）
* 開いているファイルが削除された場合、タブは閉じず「削除済み」表示にし、バッファ内容を保持する。保存すればファイルを復活できる
* タブで開いていないファイルの外部削除は、ツリーからの除去のみ行う（復元・履歴は持たない。削除のレビューは初期版の対象外とする意図的な割り切り）。ただし、未確認（未読）の新規作成ファイルが確認前に削除された場合に限り、件数のみを通知バーで知らせる（内容の復元・履歴は持たない）

### 7.3 衝突（未保存変更あり＋外部変更）

* 自動リロードしない。非モーダル通知バーで知らせる：「外部で変更されました」＋［差分を見る］［外部変更を取り込む］［編集を続行］
* ［取り込む］：未保存編集をスナップショットへ退避してからリロードする。退避内容はワンクリックで復元できる
* ［編集を続行］のまま保存する場合：「外部の変更を上書きします」と確認し、上書きされるディスク内容をスナップショットへ退避してから保存する
* 退避が物理的に取れないファイル（10MB以上・画像等、内容スナップショットを保存しない対象）に対する破壊的操作（取り込み・上書き保存・確認済み時点に戻す）は、退避不能を明示警告したうえで**既定でブロック**する（設定で「退避不能時は強い確認のうえ許可」へ切替可）
* 外部ツールがrename置換した直後にユーザーがCtrl+Sした場合も、保存前チェックは現ディスクの実内容を読んで衝突を検知し、上書きされるディスク内容をスナップショットへ退避（incoming）してから保存する（取りこぼさない。詳細は設計5.3）
* 自動マージは行わない

### 7.4 受け入れ基準（抜粋）

* 外部プロセスが1秒間に10回書き換えても、UIがフリーズせず最終内容が500ms以内に表示される
* 一時ファイル→rename方式で保存するツールによる変更も検知・反映される
* 未編集タブの自動リロードでスクロール位置が維持される
* 未保存変更があるとき外部変更が来ても、エディタ内容は変わらず通知バーが表示される
* ［取り込む］を選んだ後、退避された自分の編集を復元できる
* 100件のファイルが同時に生成・変更されても、UIがフリーズせず全件が未読化され、バッファオーバーフローが起きても再同期により取りこぼさない（多数同時変更は1秒以内に反映）
* 外部ツールがrename置換した直後にCtrl+Sしても、上書きされる外部内容が退避（incoming）され、退避なしの上書きが起きない

## 8. 差分表示と既読

### 8.1 既読ベースラインの規則

* フォルダ/ファイルを初めて開いた時点の内容をベースラインとする（全既読スタート。初回オープンで全ファイルが未読になる事態を避ける）。初回のベースライン取得は7.1の確定読み条件（静穏期間＋mtime/サイズ安定確認）を満たしてから行い、上限超過時は中途内容で確定せず保留する
* 「確認済みにする」操作でベースラインを現在内容に更新し、未読バッジを消す。内容を保存しないファイル（10MB以上・画像）はハッシュベースラインのみを更新して未読を解除する（差分表示は伴わない）。未読判定・差分の内容照合はいずれもLF正規化後の内容で行う（改行コードのみの差は未読・差分に現れない。設計8章）
* 新規作成ファイルはベースラインなしとして扱い、差分は全行追加として表示する。新規作成→外部削除→同名再作成の場合は、以前の未読/ベースラインを継承せず新規（ベースラインなし）として開始する
* pika内での自分の編集・保存はベースラインを動かさない（未読を作るのは外部変更のみ。ただし自分の編集も差分には表示される）

### 8.2 差分ビュー

* 差分は独立トグル（ON/OFF）として、ソース/分割/プレビューのいずれにも重ねて表示する（6.1）。通知バーの［差分を見る］、ツリーの右クリックからも開ける。中身は常にテキスト（ソース）ベースで、レンダリング済みプレビュー上の差分は持たない（14章）
* 比較対象：既読ベースライン vs 現在の内容（タブで開いている場合は編集バッファ、開いていない場合はディスク内容）。タブで開かずにツリーから差分表示・確認済み・確認済み時点に戻すまで完結できる
* 表示形式：インライン（unified）。行単位の赤/緑に加えて行頭に +/- 記号を併用し、変更行内の単語単位ハイライトは下線/太字も併用する（色覚多様性に配慮し色だけに依存しない）。空白区切りの単語境界が取れない文字列（日本語等）では文字単位で強調する。サイドバイサイドは初期版では持たない
* 差分ビューは読み取り専用。編集はソースモードへワンキーで切替する
* 「次の変更へ／前の変更へ」ジャンプを提供する
* 差分はテキスト（ソース）ベース。レンダリング済みプレビュー上の差分は初期版では持たない
* 差分計算はワーカースレッドで行い、2.2の段階制（10MB以上は自動オフ）に従う。差分・巻き戻しの可否はベースライン内容（object）の有無で決まり、ハッシュのみ記録のファイルでは差分・巻き戻しは非活性となる

### 8.3 確認済み操作と巻き戻し

* 「確認済みにする」：差分ビュー・通知バー・ツリー右クリック・タブバーのボタン群から実行できる
* ベースラインはディスク上の内容を対象とする。未保存変更があるタブで実行した場合は保存を促し、保存してからベースラインを更新する。差分ビューから確認済みにする場合は、差分計算に使った時点のディスクスナップショットをベースラインに採用し、確定の直前にmtime/ハッシュを再照合する。ディスクが変化していれば確認を中断して再差分を促す（ユーザーが見ていない内容をベースライン化しない）
* 「すべて確認済みにする」：全未読ファイルのベースラインを一括更新する。実行前に確認を挟む。実行開始時点の未読集合をフリーズして対象とし、更新前の各ベースラインを `baseline-replace` 退避として一括保存してワンクリックで一括取り消しできる。処理中に内容が変化したファイルは確認をスキップして未読のまま残す（並行書き込みで未確認内容をベースライン化しない）。メニューおよびフォルダ右クリック「このフォルダ配下を一括確認済みにする」からも実行できる
* 「確認済み時点に戻す」（ファイル単位の巻き戻し）：実行前に確認を挟み、巻き戻しで失われる現在内容はスナップショットへ退避する。退避が取れないファイル（10MB以上・画像等、内容を保存しない対象）では巻き戻しを提供しない（メニュー非活性。7.3の退避不能ガードに従う）

### 8.4 受け入れ基準（抜粋）

* 外部変更が複数回続いた後の差分が「前回確認時点 → 現在」の累積で表示される
* 変更行内の変わった単語だけが強調表示される
* 「確認済みにする」で未読バッジが消え、以後の差分が新ベースライン基準になる
* 巻き戻し後、退避された巻き戻し前の内容を復元できる

## 9. スナップショット

### 9.1 保存場所と方式

* 保存先はデータルート配下の `snapshots\`（ワークスペースのフォルダパスをハッシュ化してキーにする）。データルートは既定で `%LOCALAPPDATA%\pika\`、ポータブル版はexe隣の `./pika-data/`（決定規則は13章）。snapshotsフォルダ・index・内容objectはユーザー本人のみアクセスできる権限（ACL）で作成する
* ファイル単体で開いた場合・ワークスペース外のファイルは、ファイルパス単位のキーで同様に管理する（ベースライン・退避・差分・確認済みがフォルダモードと同等に機能する）。リネーム・移動でキーが変わる場合は旧キー配下の台帳と参照objectを新キーへ移送し、移送できない場合はベースラインを再取得して旧退避は旧キーに残し90日GCに委ねる（パスだけに依存せず内容ハッシュ等の安定識別子を併用する）
* 機密ファイル（既定で `.env`・`*.key`・`*.pem`・`*secret*` 等。settings.tomlのベースライン除外パターンで調整可）は内容を保存せずハッシュのみ記録する（差分・巻き戻しの対象外）。元ファイル削除後に平文コピーが残らないようにするデータ最小化のため
* ワークスペース内に `.pika` 等の管理フォルダ・ファイルを一切作らない（エージェントの作業ノイズ・Git差分ノイズを避ける）
* テキスト内容は圧縮して保存する（圧縮方式は実装裁量）
* 退避（conflict/incoming/rollback/baseline-replace）の内容objectには自己記述メタ（元relPath・kind・時刻・元index世代）を併記する。これにより索引（index.json）が破損しても、object群の走査から「復元待ちの退避一覧」を提示してから索引を再生成でき、退避＝最後の砦に到達不能にならない（設計原則1。詳細は設計7章）

### 9.2 種類と保持

| 種類 | 取得タイミング | 保持 |
|---|---|---|
| 既読ベースライン | フォルダ初回オープン時に対象テキストファイル全件、以後は「確認済みにする」のたびに更新 | ファイルごとに常に1つ |
| 退避（衝突・巻き戻し） | 7章・8章で定義した退避の発生時 | ファイルごとに最新10件、LRUで自動削除（未復元の退避は9.3の保護が優先）。「すべて確認済みにする」の `baseline-replace` 退避はこの10件枠とは別に、1回分を取消可能な単一バッチとして保持する |

* ベースライン対象（本項を全文書の正とする）：除外リスト外かつ10MB未満のテキストファイルは内容を保存する。10MB以上のテキストファイルと画像はハッシュのみ記録し、未読判定にのみ使う（差分・巻き戻しは対象外）。未読判定・差分照合に用いるハッシュはLF正規化後の内容に対して計算する
* 閾値跨ぎの挙動：内容ベースラインを持つファイルがAI追記で10MB以上になっても、既存のベースライン内容（object）は容量上限（9.3）まで保持し差分・巻き戻しの起点に使い続ける。次回「確認済みにする」でハッシュのみへ移行する。逆に10MB未満へ縮小した場合は、次回確認済みまではハッシュ比較のみ・差分非表示とし、状態変化を通知バーで知らせる（**Tauri 刷新 TBD・2026-06-21**：内容保存境界の 10MB は 2.2 第1段階と連動。第1段階閾値が sprint 6 の CM6 実測で改訂された場合は本項の数値も連動改訂する）
* 起動時の未読判定は mtime＋サイズの高速プレスクリーン→不一致のみハッシュ比較とし、巨大フォルダでも起動性能を守る。この高速パスは、mtimeが据え置かれサイズも同一の外部変更を取りこぼす既知の制限がある。確実な検知が必要な用途向けに、設定 `unread.fullHashOnStartup`（既定オフ）で全件ハッシュ照合へ切り替えられる
* 初回オープン時のベースライン取得はバックグラウンドで行い、UIをブロックしない

### 9.3 容量管理

* スナップショット全体の上限は既定500MB（設定変更可）。超過時は古い退避から削除する（ベースラインは削除対象外）。ただし未復元かつ生成から14日以内の退避（conflict/rollback/incoming/baseline-replace）は削除対象から保護し、保護分で上限を超える場合は削除せず通知バーで復元/破棄を促す。削除の適用順は「ファイルごと最新10件のLRU → 全体容量GC」とする
* フォルダを最後に開いてから90日経過したスナップショット一式は自動削除する。ただしベースラインと上記の保護対象退避は除外する
* 内容実体（object）は内容ハッシュで重複排除・共有されるため、退避やベースラインの削除に伴うobjectの物理削除は、どのベースライン（baselineHash）からも・どの退避（stash.hash）からも参照されていないことを確認してから行う（共有実体の誤削除でベースライン・差分・巻き戻しが壊れるのを防ぐ）

### 9.4 復元UI

* 退避からの復元は、対象タブの通知バー（退避直後）またはファイル右クリック「退避した内容を復元…」（時刻・種別の一覧から選択）で行う
* ファイルまたはワークスペースを閉じた際に、そのスナップショット（ベースライン・退避）を手動でパージ（破棄）できる手段を提供する（機密情報を残したくない場合向け。ファイル右クリック／メニューから）
* 高度な履歴ブラウザは持たない

### 9.5 受け入れ基準（抜粋）

* ワークスペースフォルダ内にpikaが管理ファイルを作らない
* 1000ファイル規模のフォルダの初回オープンで、ベースライン取得中もUIが操作できる
* アプリ終了中に変更されたファイルが、次回起動時に未読となり差分が正しく表示される
* `unread.fullHashOnStartup` をオンにすると、mtime据え置き・同一サイズの外部変更も次回起動時に未読として検知される（既定オフでは検知されない既知の制限がある）

## 10. 状態復元・最近使ったファイル・設定

### 10.1 状態復元

* 終了時に保存する状態：開いていたフォルダ、タブ一覧とアクティブタブ、各タブのカーソル・スクロール位置・表示モード・差分トグルのON/OFF、ツリーの展開状態・ツリーペインの収納状態、ウィンドウ位置・サイズ・最大化状態
* 引数なし起動で上記を完全復元する。引数あり起動でも、前回と同じフォルダならタブ状態を維持したうえで指定ファイルを開く
* 復元時に各タブのパスの存在を検査する：消失したファイルは「削除済み」表示で開きスナップショット/未読を紐づける、内容が別物に変わっている場合は未読として復元する、開いていたワークスペース（フォルダ）自体が消失している場合は空状態へ安全に遷移し最近使った項目からの再選択を促す（終了中にAIがファイルを削除・移動するのは中心ユースケースのため）
* 終了時に未保存変更があれば確認ダイアログ（保存／破棄／キャンセル）を出す。ホットエグジット（未保存のまま持ち越し）は初期版では持たない
* 状態はデータルート（既定 `%LOCALAPPDATA%\pika\`、ポータブル版はexe隣 `./pika-data/`。13章）配下に保存する

### 10.2 最近使ったファイル/フォルダ

* ファイル・フォルダ各20件を保持し、メニューから開ける
* Windowsタスクバーのジャンプリストにも最近使った項目を表示する

### 10.3 設定

* 形式：データルート配下の `settings.toml`（既定 `%APPDATA%\pika\settings.toml`）。pika自身で開けるコメント付き設定ファイルで、保存すると即時反映する（GUI設定画面は初期版では持たない）。settings.tomlは**ユーザー編集専用（pikaは書き戻さない）**とし、テーマの現在値・最近使った項目・ウィンドウ状態などpika側で変化する値は `state.json` に保持する（コメント・書式の破壊を避けるため）
* 設定項目：除外リスト、巨大ファイル閾値・行長閾値、レンダリング暴走ガードの閾値・タイムアウト、初回既定モード、新規ファイルのエンコーディング/改行コード、リモートリソース許可（既定オフ）、折返し既定、タブ幅・Tabキー挙動・空白/タブの可視化、フォント・サイズ、テーマ、スナップショット容量上限、スナップショットのベースライン除外パターン（機密ファイル）、起動時の全件ハッシュ照合 `unread.fullHashOnStartup`、機能トグル（Mermaid・数式・コードハイライト等）
* 不正な設定値は既定値にフォールバックし、通知バーで警告する（起動不能にしない）。settings.tomlの変更監視は文書監視と同じデバウンス・アトミック読み取りを用い、保存途中の不完全なTOMLでパースに失敗した場合は直前の有効な設定を維持して通知バー警告のみ出す（既定値への全戻しでちらつかせない）

### 10.4 受け入れ基準（抜粋）

* 再起動後、前回のフォルダ・タブ・カーソル位置・表示モードが復元される
* settings.toml を編集・保存すると、再起動なしで設定が反映される
* settings.toml が壊れていてもアプリは既定値で起動し、警告が表示される

## 11. UI構成

### 11.1 レイアウト

* 構成：メニューバー／左ペイン＝ファイルツリー（折りたたみ可）／タブバー／通知バー（必要時のみ）／メイン領域（ソース・プレビュー・分割。差分は独立トグル＝6.1）／ステータス（右下固定。フル幅の帯ではない）
* ツールバーは持たない。モード切替と「確認済みにする」はタブバー右端のコンパクトなボタン群＋ショートカットで提供する（主要機能2クリック以内の原則）
* メニューおよびツリー右クリックに「すべて確認済みにする」「（フォルダ）このフォルダ配下を一括確認済みにする」を配置する
* ステータスは画面右下隅に固定表示する（フル幅の帯ではない。視覚詳細は `docs/ui-design.md` 9章）。内容は**差分あり（未読）ファイル数**・全体の行数・文字数・カーソル位置（行・文字）。エンコーディング・改行コード・表示モードは「表示」メニューへ移す（文字コードの Reopen/Change Encoding 導線はメニュー側で維持＝5.2）
* 通知バーは非モーダルで複数キューに対応する。種類：外部変更の衝突、外部リソース参照検知、JS依存HTML検知、巨大ファイル・レンダリング制限、ファイル削除、設定エラー
* 通知バーのキュー運用：同時表示は最大3本とし、超過分は「他N件」に集約する。種類の優先順位は 衝突 ＞ 設定エラー ＞ 外部リソース参照検知 ＞ JS検知 ＞ 巨大ファイル制限 とする。同一ファイル・同一種別の通知は最新へ集約（合体）する。タブ固有の通知（衝突等）はタブ切替で表示内容も切り替わり、グローバル通知（設定エラー等）は常時表示する。各通知の自動消滅条件（解消時に消える／ユーザーが閉じるまで残る）を種類ごとに定める
* ダイアログは破壊的操作の確認（終了時・上書き・巻き戻し）に限定する

| Ctrl+\ （代替: Ctrl+Shift+E） | 分割表示 |
| Ctrl+Shift+D | 差分トグル（ON/OFF） |
| Ctrl+Enter（差分/プレビューにフォーカス時）／Ctrl+Shift+Enter（フォーカス非依存） | 確認済みにする |
| Ctrl+Alt+Enter | すべて確認済みにする |

| F5 | 全体再同期（ツリー構造・未読・差分基準・タブ内容を現ディスクへ即時再同期。監視が機能しない環境向け。12.1参照） |

* Ctrl+Enter は差分/プレビューにフォーカスがあるときのみ「確認済みにする」を発火し、エディタ編集中の誤爆を防ぐ（エディタフォーカス時は Ctrl+Shift+Enter を使う）。F8系・Ctrl+\ は他ツール/IME/JISキーボードと衝突しうるため代替割当を併記する（初期版はキーボードショートカットのカスタマイズを持たない）

### 11.3 テーマ

* ライト／ダーク／システム追従（既定：システム追従）を初期版に含める
* エディタ・ツリー・Markdownプレビューにテーマを適用する
* HTMLプレビューは文書自身のスタイルを尊重し、ダーク化による改変をしない

### 11.4 受け入れ基準（抜粋）

* マウスのみで「ファイルを開く→プレビュー→差分→確認済み」が2クリック以内ずつで完了する
* OSのテーマ切替に追従して再起動なしで配色が切り替わる
* 通知バーが出てもエディタの操作がブロックされない
* 通知が4件以上同時に発生しても最大3本＋「他N件」に収まり、同一ファイルの衝突通知は1本に集約される。タブを切り替えるとそのタブ固有の通知に表示が切り替わる
* キーボードのみで「ファイルを開く→プレビュー→差分→確認済み」が完了でき、差分の追加/削除がモノクロ（記号）でも判別でき、ハイコントラストモードでも判読できる

### 11.5 アクセシビリティ

> **【Tauri 刷新で改訂・2026-06-21・design doc 17章】** 旧版は wx ネイティブコントロールが UIA/MSAA の
> アクセシブルネームを自動供給した。全Web UI ではこれが消えるため **ARIA で再構築する**（要件11.5 を満たす
> 唯一の手段）。これを書かないと 11.5 が wx 前提で永久未達扱いになる。

* 差分表示は色だけに依存せず、追加/削除を記号（+/-）でも区別する
* ファイルツリー・タブ・モード切替・通知バーのボタン群をマウスなしで操作でき、ペイン間のフォーカス移動（F6/Shift+F6）を提供する。**F6/Shift+F6 はフロントの自前フォーカスマネージャで実装する**（design doc 17章）
* Windowsのハイコントラストモードとテキストスケーリングに追従する。**`forced-colors: active`/`prefers-contrast` で独自トークンを降格し OS の system-color に委ねる**。寸法は rem/em 基準にし WebView2 ズーム/DPI に連動する（design doc 10章）
* **全Web UI で ARIA により主要UIのアクセシブルネームを供給する**（ツリー=`role="tree"/"treeitem"`・`aria-expanded`/`aria-selected`、タブ=`role="tablist"/"tab"`、通知バー=`role="status"`/`aria-live="polite"`、ステータス=表示専用で差分あり件数を `aria-label`）。状態マーク（±/◆/取消線）は読み上げテキスト化する。プレビュー別WebView 内本文をナレーターが辿れるかは実機検証する（design doc 15章 Open 4）
* 深掘りした最適化は段階的採用とするが、上記は初期版の受け入れ基準とする
* （保全）旧要件: 「ネイティブコントロール採用（wxWidgets）の利点を活かし、ナレーター/UIA・MSAA で主要UIの
  アクセシブルネームが読まれるようにする」。← 全Web化で UIA 自動供給が消えるため上記 ARIA 再構築へ改訂。

## 12. エッジケース・エラー処理

### 12.1 ファイルシステム関連

| 状況 | 要件 |
|---|---|
| 読み取り専用属性のファイル | 開ける。保存時に「名前を付けて保存」へ誘導するか属性解除を確認する |
| アクセス権なし・他プロセスの排他ロック | リトライ後、エラー表示。クラッシュ・フリーズしない |
| シンボリックリンク／ジャンクション | たどるが、循環参照を検出してツリーの無限展開を防ぐ |
| ネットワークドライブ・UNCパス | 開ける。ファイル監視が機能しない環境では定期ポーリング（既定5秒・設定可）へフォールバックして未読・ツリーを再同期し、その旨を通知する。F5で即時の全体再スキャンも提供する |
| OneDrive等のクラウドプレースホルダ | オンデマンドファイルをベースライン内容取得の対象から除外し（9章の例外）、ユーザーが開いたときのみ読み込む。フォルダを開いただけで全ファイルがダウンロードされる事故を防ぐ |
| 開いているフォルダ自体の削除・ドライブ切断 | 通知して空状態へ安全に遷移する |
| クラッシュ耐性 | 状態ファイル・スナップショット・設定はアトミック書き込みとし、クラッシュしても破損しない。退避スナップショットは残る |
| シングルインスタンスのロック残留 | 前回異常終了後も正常に起動できる |

### 12.2 非テキストファイル

* 画像（`.png` `.jpg` `.gif` `.webp` `.bmp` `.ico`）は簡易画像ビューで表示する（表示のみ。ウィンドウフィットと等倍の切替程度。編集・変換は持たない）。デコード前にヘッダから寸法を取得し、総ピクセル数が2.2のレンダリングガード上限を超える場合はデコードせず、通知バーで「既定のアプリで開く」へ誘導する（巨大画像のデコード爆発で固まらないため）
* 画像も変更検知・未読バッジの対象とする。ベースラインはハッシュのみ記録し、内容スナップショット・差分・巻き戻しは対象外
* その他のバイナリは「対応していない形式」と表示し、「既定のアプリで開く」ボタンを提供する

### 12.3 診断ログ

* エラー・警告を `%LOCALAPPDATA%\pika\logs\` に記録する（1ファイル5MB・3世代のローテーション目安）
* ログにユーザーのファイル内容を書き込まない（ファイルパス・操作・エラー情報のみ）
* 不具合報告に使えるよう、メニューからログフォルダを開ける

## 13. 配布・インストール

* ユーザー単位インストーラー（管理者権限不要）とポータブルzipの2形態で配布する
* データの保存先（snapshots・state.json・settings.toml・logs）はデータルート配下に置く。データルートは、exe と同じフォルダに `portable.txt` マーカーがあればそのフォルダ直下の `./pika-data/`、無ければ `%LOCALAPPDATA%\pika\`（settings は `%APPDATA%\pika\`）とする。ポータブル版（`portable.txt` 付き）は持ち運び可能で実行ホストに痕跡を残さない。データルートの解決は起動最初期に1回だけ行い、全モジュールへ確定パスを渡す
* インストーラーはエクスプローラー統合（3.3）の登録・解除を行う。ポータブル版は登録しない（自己登録もしない）。アンインストール時は `HKCU\Software\Classes` 配下の登録（`OpenWithProgids`・`shell\pika`）を完全に削除し、関連付けの残骸を残さない
* アンインストール時のユーザーデータ：設定・状態・ログは削除する。snapshots（退避を含む＝最後の砦になりうる。最大500MB）はユーザー選択（既定＝残す）とし、アンインストーラーのカスタムページで選ばせる
* 状態ファイル（state.json/index.json）は `version` を持ち、未知の（新しい）version を検出した場合は読み込まず・書き戻さず・再生成もせず安全側に倒す（自動更新を持たず複数バージョンが混在しうるため、旧版が新版の状態を破壊しない）。version は破壊的変更時のみ上げる単調増加とする
* 同梱するサードパーティOSS（wxWidgets/Scintilla・md4c・差分ライブラリ・zstd・xxhash・toml11・PCRE2・Mermaid・KaTeX・コードハイライト・WebView2 Loader 等）のライセンス文・著作権表示を、インストーラー/zip および About 画面に同梱する（MIT/BSD等の再配布条件・WebView2 配布条項の遵守）
* 同梱JS/CSS（Mermaid/KaTeX/ハイライト）は外部通信を遮断し自動更新も持たないため、一度配布すると更新されない。重大な脆弱性（XSS/DoS級CVE）が出た場合はパッチ配布で同梱版を更新する運用前提とする
* 自動更新は初期版では持たない
* コード署名：初期版は**コード署名を行わない**。未署名により SmartScreen 警告（「WindowsによってPCが保護されました」）が出るため、`docs/install.md` に警告の回避手順（「詳細情報」→「実行」等）を明記し、未署名配布のリスク受容を本書に記録する（「開発フェーズで別途判断」の先送りはしない）
